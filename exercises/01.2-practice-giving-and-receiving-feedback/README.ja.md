# 01.2 練習：フィードバックの与え方と受け取り方

効果的なフィードバックは、生産的なソフトウェア開発チームの基盤です。コードの品質向上、信頼の醸成、継続的な学習の支援に役立ちます。このレッスンでは、**SBI（状況-行動-影響）モデル**と**アクティブリスニング**の技術を使って、建設的にフィードバックを与え、受け取る方法を学びます。

## 構造化されたフィードバックのためのSBIモデル

SBIモデルは、明確で敬意を持ち、実行可能なフィードバックを3つの部分に分けて伝えるのに役立ちます：

- **状況（Situation）：** 行動が起こった具体的な状況を説明します。
- **行動（Behavior）：** 判断をせずに観察可能な行動を説明します。
- **影響（Impact）：** その行動があなたやチーム、プロジェクトに与えた影響を共有します。

### 例：

「最近のログイン機能のプルリクエスト（状況）で、エラーハンドリングがネットワーク障害をカバーしていないことに気づきました（行動）。これにより、アプリが静かに失敗した場合にユーザーが混乱する可能性があります（影響）。」

「あなた」ではなく**「私」メッセージ**を使うことで、非難を避け、会話を前向きに保つことができます。例えば、「ここにエラーハンドリングを追加するとユーザー体験が向上すると思います」と言い、「あなたはエラーハンドリングを忘れた」とは言いません。

## アクティブリスニングでフィードバックを受け取る

フィードバックを受け取る際は、以下のステップを実践してください：

- **返答する前に一旦止まる。** フィードバックを処理する時間を取ります。
- **聞いたことを言い換える。** 例えば、「つまり、エラーハンドリングをもっと堅牢にすべきだと言っているのですね？」
- **明確化の質問をする。** 例えば、「ネットワークエラーの扱い方について、どのように改善すればよいか提案してもらえますか？」
- **建設的に応答する。** フィードバックをくれた人に感謝し、防御的な反応を避けます。

## 練習シナリオ：GitHubプルリクエストのコメント

GitHubでプルリクエストをレビューしていると想像してください。以下はフィードバックのやり取りの例です：

```text
レビュアー：「最近のログイン機能のPRで、エラーハンドリングがネットワーク障害をカバーしていないことに気づきました。これにより、アプリが静かに失敗した場合にユーザーが混乱する可能性があります。」
開発者：「指摘ありがとうございます。より包括的なエラーハンドリングを追加します。」
```

レビュアーのコメントをSBIモデルと「私」メッセージを使って書き直し、アクティブリスニングを用いた建設的な応答を作成してみてください。
